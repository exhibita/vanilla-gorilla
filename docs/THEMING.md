# Theming Vanilla Gorilla

Vanilla Gorilla doesn't have a plugin-style theme system with a `--theme=` flag or a `themes/` directory you drop in and select. There is exactly one active source tree, `src/`, and its templates, skeletons, CSS, and images collectively **are** the theme. Content (blog posts, pages, galleries) lives in that same tree, right alongside the theme files.

This means "creating a new theme" is really "replacing the presentation layer inside `src/`" and "converting an existing site" is really "reshaping that site's HTML into the handful of files `build.js` knows how to process." Both are covered below.

Before touching anything, it helps to know exactly what `build.js` depends on, because those dependencies are hardcoded (not configurable) and breaking them silently disables features like pagination, RSS, and post navigation.

## 1. How the build actually works (the parts that matter for theming)

There's no templating language here — no `{{ variable }}`, no `{% for %}`, no Handlebars/EJS/Nunjucks. `build.js` (Node.js + `cheerio` + `marked` + `gray-matter`) does three things to turn `src/` into `dist/`:

1. **Expands SSI-style includes.** Any file containing `<!--#include file="templates/header.html" -->` gets that comment replaced with the raw contents of the referenced file (`build.js:92-118`). This is the *entire* "reusable partial" mechanism.
2. **Merges Markdown into a skeleton.** A `.md` file's front matter picks a skeleton (`src/skeletons/<name>-skeleton.html`), and `build.js` uses cheerio to inject values into specific CSS-selector "slots" that the skeleton must contain (`build.js:265-300`).
3. **Rewrites relative links by folder depth** so every page can use root-relative-looking `./` links regardless of how deeply nested it is (`build.js:65-89`).

Everything else — copying images, generating the blog/gallery index pages, RSS, sitemap, prev/next navigation — is built on top of those three mechanisms and a fixed set of selector names.

### The selector contract

If you replace the theme's HTML, these exact classes/IDs must still exist wherever `build.js` expects them, or the corresponding feature silently stops working (usually with a console warning, not a build failure):

| Selector | Lives in | Purpose |
|---|---|---|
| `<title>` | any skeleton | Set to `"<front matter title> - <siteTitle>"` |
| `.entry-title` | skeleton | Filled with the page/post title |
| `.entry-content` | skeleton | Filled with the rendered Markdown body |
| `time.entry-date` | skeleton | Given a `datetime` attribute + display text; if front matter has no `date`, the closest `.posted-on` ancestor is removed entirely |
| `.entry-footer` | skeleton | Removed entirely on non-blog pages (pages that don't resolve to a `blog/<slug>/` path) |
| `meta[name="description"]` | skeleton `<head>` | Replaced from front matter `description` |
| `#post-nav-placeholder` | `blog-skeleton.html` | Filled with prev/next links after all posts are compiled (`build.js:509-540`) |
| `#blog-post-list` | `templates/blog-index.html` | Filled with the generated list of blog post `<article>` entries |
| `#gallery-list` | `templates/gallery-index.html` | Filled with the generated list of gallery entries |

If a theme doesn't want blog dates, footers, or navigation, the simplest option is to omit those elements from the skeleton — `build.js` degrades gracefully (it just skips the injection) rather than erroring.

### Files that make up "the theme"

```
src/
├── templates/
│   ├── header.html        # opens <html><head>...<body>...<main>, left open on purpose
│   ├── footer.html         # closes what header.html opened, adds <footer>
│   ├── blog-index.html     # wrapper page for the generated, paginated blog listing
│   ├── gallery-index.html  # wrapper page for the generated, paginated gallery listing
│   └── recent-posts.html   # AUTO-GENERATED — do not hand-edit, build.js overwrites it every run
├── skeletons/
│   ├── blog-skeleton.html     # shell used for any Markdown page (front matter `skeleton: blog`, the default)
│   └── gallery-skeleton.html  # example static gallery page shell w/ CSS-only lightbox markup
├── css/
│   └── style.css           # hand-authored; style.min.css is generated by `node minify.js`, never edit it directly
├── images/
│   ├── logo.png
│   └── ...
└── favicon.ico
```

`header.html` and `footer.html` are two halves of one document — `header.html` ends mid-`<main><div class="container">` and `footer.html` starts by closing those tags. Any page that includes both, in order, gets a complete valid HTML document.

## 2. Building a new theme from scratch

Recommended order, so you can check the build after each step rather than debugging everything at once:

1. **Copy the shipped theme as a starting point** rather than deleting it outright: `git mv` or duplicate `src/templates`, `src/skeletons`, `src/css/style.css`, `src/images/logo.png`, `src/favicon.ico` somewhere safe (or just work on a branch), so you can diff against a known-good build.

2. **Redesign `templates/header.html` and `templates/footer.html`.**
   - Keep the split point consistent: header opens everything through the start of your main content wrapper, footer closes it.
   - Keep the stylesheet `<link>` (`./css/style.min.css`) and favicon `<link>` pointing at whatever your new asset paths are.
   - Nav links should use `./`-relative paths (e.g. `./blog/`) — the depth-rewriting step in `build.js` (`adjustDocumentLinks`, `build.js:65-89`) automatically turns these into `../blog/`, `../../blog/`, etc. on nested pages. Don't hardcode `../` yourself.
   - `templates/recent-posts.html` is regenerated by the build every run (`build.js:487-506`); if your new header/footer/homepage wants a "recent posts" widget, include it via `<!--#include file="templates/recent-posts.html" -->` and let the build populate it — don't hand-write its contents.

3. **Rebuild `src/css/style.css`.** The shipped stylesheet defines its whole visual language as CSS custom properties at the top (`src/css/style.css:8-35`):
   ```css
   :root {
     --bg-primary: #f8f6f2;
     --text-main: #2b2927;
     --color-primary: #596e57;
     --font-sans: 'Outfit', sans-serif;
     --font-serif: 'Playfair Display', serif;
     --max-width: 960px;
     --radius-md: 12px;
     /* ...shadows, secondary/accent colors, transitions... */
   }
   ```
   You can keep this variable-driven structure (recommended — it's what the rest of the stylesheet's components reference) or replace it wholesale. Either way, never hand-edit `src/css/style.min.css` — it's a generated artifact; run `node minify.js` (or `npm run build`) to regenerate it from `style.css`.

4. **Rework the skeletons.** At minimum, `src/skeletons/blog-skeleton.html` needs the selector contract from the table above. A minimal valid skeleton is just:
   ```html
   <!--#include file="templates/header.html" -->
   <article class="post hentry">
       <header class="entry-header">
           <h1 class="entry-title"></h1>
           <div class="entry-meta"><span class="posted-on"><time class="entry-date"></time></span></div>
       </header>
       <div class="entry-content"></div>
       <footer class="entry-footer">
           <nav><div id="post-nav-placeholder"></div></nav>
       </footer>
   </article>
   <!--#include file="templates/footer.html" -->
   ```
   Add more skeletons for other content types (docs, products, landing pages) by dropping `src/skeletons/<name>-skeleton.html` files — front matter `skeleton: <name>` on any `.md` page selects them (`build.js:265-266`), and `create-page.js`/`create-md` auto-discover them for scaffolding (`create-page.js:30-37`).

5. **Rebuild `templates/blog-index.html` and `templates/gallery-index.html`** with your new header/footer and page furniture, keeping the `#blog-post-list` / `#gallery-list` container elements intact so the generated listings land in the right place.

6. **Swap branding assets** — replace `src/images/logo.png`, `src/favicon.ico`, and any other images referenced from header/footer, updating `alt` text as needed.

7. **Verify.** Run:
   ```
   npm run build
   ```
   then open `dist/index.html`, `dist/blog/`, and a couple of individual blog posts and check: styling loads, nav links resolve correctly at every depth, blog listing/pagination renders, prev/next post links appear, RSS (`dist/rss.xml`) and sitemap (`dist/sitemap.xml`) still generate without warnings in the console output. `npm run watch` is useful while iterating — it rebuilds automatically, though note that **changes to `src/skeletons/*` only take effect on already-existing content the next time that content is recompiled**, not retroactively on watch (skeleton edits don't trigger a rebuild by themselves — see `build.js:698-700`); do a full `npm run build` after finishing skeleton changes.

### On "multiple installable themes"

There's currently no mechanism to keep several themes side by side and pick one at build time (no `src/themes/<name>/` convention, no config key for it). If you want that, it's a real code change to `build.js`, not just an asset swap — you'd parameterize `SRC_DIR`/the templates-and-skeletons lookup paths (`build.js:13`, `266`, `374`, `766`) behind a `config.json` key or `--theme` CLI flag. Worth calling out explicitly if you're mentioning this to contributors: today, "changing the theme" means editing the one active `src/` tree in place, not selecting between packages.

## 3. Converting an existing HTML website into Vanilla Gorilla

The general approach: pull the repeated chrome (nav, head boilerplate, footer) out into `templates/header.html`/`footer.html`, decide per content-type whether pages will be static HTML or Markdown-authored, and adjust every link to be `./`-relative.

### Step 0 - Consider using your AI Agent to do the work for you

Vanilla Gorilla has recently added a new skill that will guide and assist you through applying an existing design into your new site. You have the choice of pointing to an existing sketch, local static file(s), a live URL, or a Figma sketch. This skill is normally launched by inference in conversation with your Agent but can also be called directly using the name `design-apply` in the manner that is appropriate to your particular Agent.

### Step 1 — Extract header/footer

Take any representative page from the existing site. Find the split point — usually right after the opening `<nav>`/site-chrome and right before the closing `</main>`/footer markup. Put everything above that point into `src/templates/header.html`, everything below it into `src/templates/footer.html`. Keep both halves *unclosed*/*unopened* respectively, exactly like the shipped theme does.

Replace hardcoded links in the extracted nav (e.g. `href="/blog"` or `href="https://mysite.com/blog"`) with root-relative `./`-prefixed links (`href="./blog/"`). The build's depth-rewriter needs the `./` prefix to know which links to adjust — absolute URLs and `#anchors` are left alone.

### Step 2 — Decide: static HTML pass-through, or Markdown + skeleton?

For each existing page type, pick one:

- **Static HTML pass-through** — for one-off pages (an About page, a legal page, a landing page) that don't share a repeating structure with other content. Just take the page's unique body content, wrap it with the two include directives:
  ```html
  <!--#include file="templates/header.html" -->
  <!-- ...this page's unique content, links converted to ./-relative... -->
  <!--#include file="templates/footer.html" -->
  ```
  and place it at `src/<slug>/index.html`. `compileHtmlFile` passes it through basically unchanged (includes expanded, links depth-adjusted) — no front matter, no skeleton involved (`build.js:199-234`).

- **Markdown + skeleton** — for any repeating content type (blog posts, docs articles, changelog entries, portfolio items). Build one `src/skeletons/<type>-skeleton.html` shell with the selector contract from section 1, then convert each existing page's body content to Markdown and give it front matter:
  ```markdown
  ---
  title: Original Page Title
  date: 2024-03-15
  description: The meta description this page used to have.
  skeleton: blog
  ---

  Body content, converted from HTML to Markdown...
  ```
  Place it at `src/<content-type>/<slug>/index.md`. `compileMarkdownFile` renders the Markdown, injects it and the front matter into the skeleton's slots, and writes `dist/<content-type>/<slug>/index.html` (`build.js:237-328`).

  Converting HTML body content to Markdown: for simple prose/images/lists, this is usually just hand-conversion (or running the old page through any HTML→Markdown converter first, then hand-cleaning it — `marked` renders standard Markdown back to HTML on build, so keep any HTML you truly need to preserve inline in the `.md` file; `marked` passes raw HTML through untouched).

### Step 3 — Preserve URL structure

Vanilla Gorilla always compiles to `<slug>/index.html`, servable as `/<slug>/`. If the old site's URLs were e.g. `/about.html` or `/blog/my-post.html`, decide whether to keep matching pretty-URL slugs (`src/about/index.html`, `src/blog/my-post/index.md`) or accept the URL changing. If preserving exact old URLs matters (SEO), you'll need redirects — outside the scope of the compiler itself (handle at your hosting layer, e.g. GitHub Pages custom 404 or CloudFront function).

Two path shapes are load-bearing if you want the blog/gallery features:
- Blog posts must be at exactly `blog/<slug>/index.{html,md}` — `blog/author/...` is explicitly excluded (`build.js:122-127`).
- Galleries must be at exactly `gallery/<slug>/index.{html,md}` (`build.js:43-48`).

Anything else (images, PDFs, fonts, arbitrary static files) is copied byte-for-byte to the same relative path (`build.js:330-344`) — no conversion needed, just drop them in the equivalent `src/` location.

### Step 4 — Convert asset references

Any link, `<img src>`, or CSS `url(...)` that used `./`-relative paths in the original site works as-is (the depth-rewriter will adjust them per page). Absolute paths (`/images/foo.png`) or absolute URLs pointing at the old domain need to be converted to `./images/foo.png`-style relative paths, with the actual files copied into `src/images/` (or wherever you choose).

### Step 5 — Rebuild global bits

- Update `siteUrl`, `siteTitle`, `siteDescription` — either via a `config.json` at the repo root (not committed by default) or `SITE_URL`/`SITE_TITLE`/`SITE_DESCRIPTION` environment variables (`build.js:16-33`). These feed the RSS feed, sitemap, and `<title>`/meta description defaults.
- If the old site had RSS/sitemap already, you don't need to port them — `npm run build` regenerates `dist/rss.xml` (from blog posts with a `date` in front matter) and `dist/sitemap.xml` (walked from everything compiled to `dist/`) automatically.

### Step 6 — Verify

```
npm run build
```
Check the console for warnings — missing includes, missing `title` front matter, skeleton-not-found, and skipped-due-to-collision (`.md`/`.html` sibling at the same destination) all print warnings rather than failing the build outright, so a clean-looking build can still have silently dropped pages. Then spot-check `dist/` output for a few pages at different nesting depths to confirm links resolve.

## Quick reference: relevant files

| File | Role |
|---|---|
| `build.js` | The entire compiler — includes, skeleton merge, link rewriting, blog/gallery/RSS/sitemap generation |
| `minify.js` | Regex-based CSS minifier: `src/css/style.css` → `src/css/style.min.css` |
| `create-page.js` | `npm run create-page` / `create-md` scaffolding CLI, auto-discovers skeletons from `src/skeletons/*-skeleton.html` |
| `src/templates/*.html` | Global chrome partials, included via `<!--#include file="..." -->` |
| `src/skeletons/*-skeleton.html` | Per-content-type shells merged with Markdown front matter + body |
| `src/css/style.css` | Hand-authored theme styles (edit this, not the `.min.css`) |
| `config.json` (optional, not committed) | Overrides `siteUrl`, `siteTitle`, `siteDescription`, `rssItemLimit`, `postsPerPage` |
